# Agent Instructions — tree-sitter-groovy

Guidance for AI agents (and humans) working on this repo. **Read this before making changes.**

## What this is

A [tree-sitter](https://tree-sitter.github.io/) grammar for **Groovy**, forked from
[`murtaza64/tree-sitter-groovy`](https://github.com/murtaza64/tree-sitter-groovy) and
extended to parse the full breadth of Groovy found in **Jenkins pipelines**, **shared
pipeline libraries**, and general Groovy source. It is the source of the WASM bundled by
`bootswithdefer/prettier-plugin-jenkinsfile`.

## Two coupled repos

| Repo | Role |
|------|------|
| `bootswithdefer/tree-sitter-groovy` (this repo) | The grammar. Source of truth. |
| `bootswithdefer/prettier-plugin-jenkinsfile` | Prettier plugin that bundles this grammar compiled to `tree-sitter-groovy.wasm`. |

**When you change the grammar, the plugin does not update automatically.** Rebuild the
WASM and copy it into the plugin (see [Propagating to the plugin](#propagating-to-the-plugin)).

## Layout

```
grammar.js          The grammar definition. This is the ONLY hand-edited grammar source.
src/                GENERATED by `tree-sitter generate` (parser.c, grammar.json,
                    node-types.json). Committed, but never hand-edit — regenerate.
queries/            Editor queries (highlights, folds, indents, injections, locals).
                    These reference node types by name and BREAK if a node type is removed.
test/corpus/        Corpus tests (`*.test`), one topical file per language area.
scripts/coverage.sh Enforces 100% named-node-type coverage.
scripts/test.sh     Runs `tree-sitter test`.
```

## Non-negotiable diligence rules

Treat every one of these as a merge blocker. Do not hand off work that violates them.

### 1. Zero warnings from `tree-sitter generate`

After any `grammar.js` change, run `tree-sitter generate` and read the output.

- **`Warning: unnecessary conflicts`** — a declared entry in the `conflicts` array is no
  longer needed. **Remove it.** Do not leave unnecessary conflicts sitting in the array;
  they hide real ambiguity and rot over time. Remove the flagged pair, regenerate, and
  confirm the warning is gone and no *new* unresolved-conflict error appears.
- Only add a `conflicts` entry when `generate` reports an actual **unresolvable
  conflict** (an error, not a warning). Prefer fixing precedence/structure over adding a
  conflict when practical.
- A clean `generate` prints no `Warning:` and no `Error:` lines. Anything else is unfinished work.

### 2. 100% named-node-type coverage

`scripts/coverage.sh` asserts every named node type in `src/node-types.json` is exercised
by at least one corpus test. It is enforced by a prek hook.

- **Add a corpus test for every new named node type** you introduce (e.g. adding a
  `named_argument` rule requires a test that produces a `(named_argument …)` node).
- If you *remove* a node type, the coverage set shrinks automatically — but you must also
  remove/adjust the corpus tests and queries that referenced it (see rule 3).
- Never delete a test to make coverage or the suite pass. Fix the grammar or the test.

### 3. Removing or renaming a node type is a breaking change — sweep everything

When a node type stops being produced (e.g. collapsing a structured rule into a token),
grep the whole repo and fix every reference:

- **`queries/*.scm`** — a query referencing a gone node type fails at load time with
  `Query error: Invalid node type "<name>"`. This breaks editors/highlighters even though
  `tree-sitter test` still passes. Always run the query validation (below).
- **`test/corpus/*.test`** — update expected trees.
- **Bindings / docs** — grep for the name across `*.md`, `*.scm`, `*.js`, `*.json`.

### 4. All corpus tests pass, always

`tree-sitter test` must be 100% (`failed parses: 0`). Use `--update` only to regenerate
expected trees you have *verified* are correct — never to paper over an unexpected change.
After an `--update`, `git diff` the test files and confirm only the intended trees changed.

### 5. Validate against real-world Groovy, not just the corpus

The corpus is necessary but not sufficient. Before shipping a grammar change, parse a broad
sample of real Jenkinsfiles / pipeline-library files and confirm zero parse errors. When
checking parse cleanliness, grep for **both `ERROR` and `MISSING`** nodes — a `MISSING`
node is still a parse failure (tree-sitter inserted a token to recover) and `grep ERROR`
alone will miss it.

## Development loop

```bash
# 1. Edit grammar.js
# 2. Regenerate + read warnings (rule 1)
tree-sitter generate            # must be warning- and error-free

# 3. Run the corpus (rule 4)
tree-sitter test                # failed parses: 0

# 4. Coverage (rule 2)
./scripts/coverage.sh           # 100% named-node-type coverage

# 5. Validate queries against the grammar (rule 3)
printf '/** d */\ndef f(){}\n' > /tmp/q.groovy
for q in highlights folds indents injections locals; do
  tree-sitter query queries/$q.scm /tmp/q.groovy >/dev/null && echo "$q OK"
done

# 6. Real-world check (rule 5) — parse a sample, look for ERROR|MISSING
tree-sitter parse path/to/Jenkinsfile | grep -E 'ERROR|MISSING'
```

The `.pre-commit-config.yaml` runs generate + test + coverage on commit via prek
(`prek install`). If a hook modifies files (e.g. `generate` refreshes `src/`), re-stage and
re-commit.

## Test organization

- One topical corpus file per language area: `declaration.test`, `closure.test`,
  `annotation.test`, `string.test`, `map.test`, `comment.test`, `general_groovy.test`, etc.
- **`jenkinsfile.test` is only for genuinely Jenkinsfile-specific constructs** (`pipeline {}`,
  `stage`, `steps`, pipeline-library steps). General Groovy goes in the topical files —
  do not let `jenkinsfile.test` become a grab-bag.

## Grammar conventions worth knowing

- **Groovydoc** (`/** … */`) is a single permissive **token** (`groovy_doc`) in `extras`.
  It intentionally does *not* sub-parse `@param`/`@throws`; keep it a token so arbitrary
  groovydoc content can never cause a parse error.
- **Identifier vs type ambiguity**: capitalized/mixed-case names (`TF_VAR_git_hash`) lex as
  `_type_identifier` (via `TYPE_REGEX`, longest-match). Reference/assignment positions that
  should accept such names must include `_type_identifier`, not just `identifier`.

## Propagating to the plugin

Grammar changes only reach users through the plugin's bundled WASM:

```bash
tree-sitter build --wasm -o tree-sitter-groovy.wasm .
cp tree-sitter-groovy.wasm ../prettier-plugin-jenkinsfile/tree-sitter-groovy.wasm
# then release the plugin (its own AGENTS.md / CHANGELOG)
```

## Release process

This project follows [Semantic Versioning](https://semver.org/) and
[Keep a Changelog](https://keepachangelog.com/). The npm package is published on a
GitHub **release** (see `.github/workflows/publish.yml`).

1. Ensure rules 1–5 all pass.
2. Bump the version in **all four** manifests: `package.json`, `tree-sitter.json`,
   `pyproject.toml`, `Cargo.toml`.
3. Add a `CHANGELOG.md` entry (and update the compare-link footer). Call out **Removed**
   node types explicitly — they are breaking for query consumers.
4. Commit, tag `vX.Y.Z`, and publish the GitHub release.
